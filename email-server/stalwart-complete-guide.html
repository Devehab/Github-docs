    إنشاء سيرفر Hetzner من جديد

    1. احذف السيرفر القديم إذا بدك بداية نظيفة (Cloud Console → Servers → Delete).
    2. أنشئ سيرفر جديد: CAX21 (4 vCPU / 8GB / 80GB) · Ubuntu 24.04 · موقع أوروبي · مع مفتاح SSH.
    3. بعد الإنشاء: Server → Networking → Reverse DNS (PTR) ← للـIPv4 اكتب mail.techkahwa.net ← احفظ.
    وين بالضبط

    الـPTR مو سجل بتضيفه بـCloudflare — بتعبّي خانة جاهزة بلوحة Hetzner جنب الـIP. القيمة = اسم السيرفر mail.techkahwa.net (اسم فقط، مو IP).

    نقطة فحص ١
    • السيرفر شغّال وأخدت الـIP تبعه.
    • حطيت الـPTR بلوحة Hetzner = mail.techkahwa.net.
    • قدرت تعمل ssh root@YOUR_IP ودخلت.
    إذا ما دخلت SSH: تأكد من مفتاح SSH ومن الـIP. لا تكمّل قبل ما تدخل.

    سجل DNS الأساسي (قبل التنصيب)

    لازم اسم السيرفر يأشّر على الـIP قبل ما نطلب الشهادة — وإلا بتفشل.

    بـCloudflare
    • الدومين techkahwa.net → DNS → Add record
    • Type: A · Name: mail · IPv4: YOUR_SERVER_IP
    • Proxy status: DNS only (سحابة رمادية) — مو برتقالية
    الخطأ اللي وقّعك فعلياً — تأكد منه بالذات

    بإعدادك السابق سجل mail كان يأشّر على IP غلط (95.217.238.7) والسحابة برتقالية — عشان هيك SSL Labs شاف عناوين Cloudflare، والشهادة فشلت، والدخول للّوحة فشل (كنت توصل مكان تاني مو سيرفرك).

    • الـIP الصحيح لسيرفرك: 91.98.194.24
    • سجل mail = A → 91.98.194.24 بالضبط
    • السحابة رمادية (DNS only) — تأكد بعينك

    لو الاسم بيأشّر على IP غلط، كل شي بينهار: لا شهادة، لا MX، لا دخول للّوحة. هاي أول شي تتأكد منه.

    ليش رمادي إجباري

    السحابة البرتقالية (Proxy) بتمرّر بس HTTP/HTTPS على منافذ محددة، وبتخفي الـIP الحقيقي — بتكسر منافذ البريد (25/587/993) وبتمنع Let's Encrypt من الوصول. هاد بالضبط سبب خطأ الـ408 اللي صار معك.

    تحقق من الـDNS
    dig +short A mail.techkahwa.net       # لازم يرجّع IP سيرفرك (91.98.194.24)
dig +short -x YOUR_SERVER_IP          # لازم يرجّع mail.techkahwa.net (PTR)

# لو ظهرت عناوين غريبة (104.x أو 172.x) = السحابة برتقالية، طفّيها
    لو ما بتقدر تدخل اللوحة أصلاً

    طول ما سجل mail غلط، ادخل اللوحة بالـIP المباشر بدل الاسم:

    http://91.98.194.24:8080/admin

    نقطة فحص ٢
    • dig A بيرجّع IP سيرفرك بالضبط (91.98.194.24) — مو أي IP تاني.
    • dig -x بيرجّع mail.techkahwa.net.
    • سحابة سجل mail بـCloudflare رمادية.
    إذا رجّع IP غلط (مثل 95.x أو 104.x أو 172.x): عدّل سجل A ليأشّر على 91.98.194.24 وطفّي السحابة. هاي أهم نقطة بالدليل كله — لا تكمّل قبل ما الـA يرجّع IP سيرفرك الصحيح.

    في طبقتين حماية — بس اختار وحدة

    هاد اللي عطّلك سابقاً. في طبقتين منفصلتين، ولو استعملت الاثنين بقواعد مختلفة بينحجب المنفذ عند أي وحدة:

    • Hetzner Cloud Firewall — على مستوى شبكة Hetzner (من الـConsole). الموصى به.
    • ufw — على السيرفر نفسه (بالأوامر). بديل.

    اختار وحدة بس. المشكلة سابقاً إنك فتحت 993 بـufw بس Hetzner كان حاجبه — لأنه المنفذ لازم مفتوح عند الجدار الفعّال. الحل: جدار واحد يكون المرجع، مش الاثنين.

    افهم المنافذ أول: مين بيستعملها وبأي اتجاه

    كل منفذ إله اتجاهين منفصلين: وارد (مين بيتصل بسيرفرك) وصادر (سيرفرك بيتصل بمين). أغلب المنافذ بتحتاج وارد بس. منفذ واحد بس بيحتاج الاتجاهين — وهو سبب مشكلة «ما عم برسل».

    المنفذمين بيستعملهالاتجاه المطلوبضروري؟
    25سيرفر ↔ سيرفر (Gmail/Outlook يوصّلولك · وسيرفرك يوصّلهم)وارد + صادر✅ جوهري — الصادر هو المحجوب عند Hetzner
    465عملاؤك يرسلوا (Thunderbird/موبايل → سيرفرك) — SSL مباشروارد بس✅ الأساسي للإرسال من العملاء
    587عملاؤك يرسلوا — STARTTLS (بديل 465)وارد بساختياري — 465 بيغني عنه
    993عملاؤك يستقبلوا/يزامنوا — IMAP (SSL)وارد بس✅ الأساسي للاستقبال
    995عملاؤك يستقبلوا — POP3 (SSL)وارد بس⭐ ضروري لربط Gmail (POP3)
    143IMAP بـSTARTTLS (بديل 993)وارد بساختياري — 993 بيغني عنه
    4190إدارة قواعد الفرز (Sieve) من العميلوارد بسمفيد
    443اللوحة + webmail + JMAP + الشهادةوارد بس✅ جوهري
    80تحويل HTTP + شهادة احتياطيوارد بسمفيد
    22SSH (إدارتك للسيرفر)وارد بس✅
    الخلاصة اللي بتفك الحيرة
    • الإرسال لبرّا (لـGmail) = صادر 25 حصراً. ما إله بديل. هاد المحجوب.
    • 465/587 ما لهن علاقة بالإرسال لبرّا — هنن بس باب عملائك يوصّلولك رسايلهم. 465 كافي، انسَ 587.
    • 993/995 = بابين عملائك يستقبلوا. 993 للبرامج، 995 لربط Gmail.

    حجب Hetzner التلقائي (اللي وقّعك) + كيف تطلب الفك

    Hetzner بيحجب 25 · 465 · 587 الصادرة تلقائياً

    على الحسابات/السيرفرات الجديدة، Hetzner بيحجب منافذ البريد الصادرة (25, 465, 587) افتراضياً كمكافحة سبام — بدون ما يعلمك. عشان هيك بتقدر تستقبل بس ما بتقدر ترسل لبرّا.

    كيف تطلب الفك (الرابط المباشر):

    • روح على console.hetzner.com/limits
    • اضغط Request change → Unblock Ports
    • اكتب رسالة مختصرة إنك تشغّل mail server شرعي (نموذج بخطوة ١٠)
    • بعض الحسابات بينفك تلقائياً خلال دقائق؛ غيرها بيراجعوه يدوي.
    مهم — 587 وفحص المنافذ بيجوا لاحقاً

    بخطوة ٣ هون بس بتفتح المنافذ بالجدار الناري (تحضير). المنفذ ما بيصير شغّال فعلياً إلا بعد ما تكمّل المعالج + تضبط Cloudflare/DNS + تعيد التشغيل (خطوات ٦–٧). عشان هيك تفعيل 587 وفحص المنافذ الفعلي موجودين بعد خطوة ٧ — لأنه فحصهم الآن بيطلّع كلهم «مسكّرين» وهنن لسا ما فتحوا.

    الطبقة ١ — Hetzner Cloud Firewall (من الـConsole)

    Cloud Console → Firewalls
    • Inbound: أضف قاعدة لكل منفذ من الجدول فوق: 22, 25, 80, 143, 443, 465, 587, 993, 995, 4190, 8080 (TCP · Any IPv4 · Any IPv6).
    • Outbound: خليه فاضي (Nothing) — لما يكون فاضي معناه كل الصادر مسموح تلقائياً. ما تلمسه.
    مهم عن الـOutbound ومنفذ 25

    مو لازم تفتح شي بالـOutbound — الفاضي = سماح كامل. بس انتبه: حجب منفذ 25 الصادر عند Hetzner مو من هالجدار، هو على مستوى شبكة Hetzner (upstream). ما بينحل من هون ولا من ufw — بس بطلب فك حجب (خطوة الاختبار الحاسمة تحت).

    مهم: اختار جدار واحد — مش الاثنين

    ما بتحتاج الطبقتين مع بعض. أي واحد منهن بيكفي لوحده. وجود الاثنين بقواعد مختلفة = مصدر البلبلة اللي صارت معك (فتحت 993 بـufw بس ضل محجوب لأنه Hetzner كان حاجبه).

    توصيتي: اعتمد على Hetzner Cloud Firewall بس (بيشتغل على مستوى الشبكة قبل ما توصل الحزمة للسيرفر — أقوى وأوضح)، واطفي ufw.

    لو اخترت Hetzner Cloud Firewall → اطفي ufw

    الترتيب مهم — حتى ما تنقفل برّا

    أول ظبّط Hetzner Cloud Firewall كامل وتأكد إنه منفذ 22 (SSH) مفتوح فيه، وبعدها بس اطفي ufw. لو عكست الترتيب وHetzner ما فيه 22، بتنقفل برّا السيرفر.

# تنظيف كامل اختياري: ufw reset ثم ufw disable

    تبسيط (موصى به)

    حتى ما تتلخبط بين جدارين: خلّي واحد بس هو المصدر. الأنظف تعتمد على Hetzner Cloud Firewall (أوضح من اللوحة) وتترك ufw مفتوح/مطفي، أو العكس. وجود الاثنين بقواعد مختلفة = مصدر رئيسي للأخطاء.

    الاتجاه الصادر (Outbound) — اقرأ قبل ما تعمل قواعد

    الأبسط والأأمن: لا تعمل قواعد Outbound أبداً

    الافتراضي (ufw = allow outgoing، وHetzner Outbound = فاضي) معناه كل الصادر مسموح. سيرفرك أصلاً بيطلع على كل المنافذ اللي يحتاجها. خليه هيك. إضافة قواعد Outbound يدوياً بتقلبها لـ«اسمح بس بالمذكور» — وأي منفذ نسيته بينحجب ويكسر السيرفر.

    لو أصرّيت على تحكّم صريح — حصر 80/443/25 بس خطير وناقص

    حصر الصادر على 80/443/25 فقط بيكسر السيرفر — ناقص منافذ حيوية، أهمها 53 (DNS): بدونه السيرفر ما بيلاقي عنوان Gmail أصلاً، فكل البريد بيتوقف. القائمة الصادرة الصحيحة الكاملة:

    • 53 UDP + TCP — DNS (حيوي — بدونه ما في بريد ولا شهادة)
    • 25 TCP — تسليم البريد للسيرفرات الخارجية
    • 465 + 587 TCP — تسليم لبعض السيرفرات + relay إذا استعملت
    • 80 + 443 TCP — ACME/شهادات + تحديثات النظام + MTA-STS
    • 123 UDP — NTP (ضبط الوقت — مهم للشهادات والتوقيع)
    ufw — قواعد صادرة صريحة (فقط لو أصرّيت)
    # الافتراضي أسهل؛ نفّذ هذا فقط لو بدك تحكّم صارم
ufw default deny outgoing
ufw allow out 53         # DNS (UDP+TCP)
ufw allow out 123/udp    # NTP
ufw allow out 25/tcp
ufw allow out 465/tcp
ufw allow out 587/tcp
ufw allow out 80/tcp
ufw allow out 443/tcp
ufw reload
    وعلى Hetzner Cloud Firewall

    لو ضفت قواعد Outbound بلوحة Hetzner، لازم نفس القائمة فوق (خصوصاً 53 و25). توصيتي: خلّي Outbound فاضي (Nothing) بـHetzner، وإذا لازم تحكّم صارم اعمله من ufw بس — حتى ما تكرّر القواعد بمكانين.

    تذكير: حجب 25 الصادر مش من هون

    حتى لو سمحت بـ25 الصادر بجدارَيك، Hetzner بيحجبه upstream على الشبكة. سماحك بالجدار ضروري بس مو كافي — لازم كمان طلب فك الحجب من console.hetzner.com/limits.

    نقطة فحص ٣ — تحضير الجدار فقط
    • كل المنافذ مفتوحة Inbound بالجدار (22, 25, 80, 143, 443, 465, 587, 993, 995, 4190, 8080).
    • الـOutbound فاضي (Nothing) — كل الصادر مسموح.
    • اخترت جدار واحد (Hetzner موصى به) وطفّيت التاني.
    ملاحظة: فتح المنفذ بالجدار مش معناه إنه شغّال — Stalwart لسا ما فتحه. الفحص الفعلي للمنافذ بعد خطوة ٧ (بعد المعالج والـrestart).

    فحص الشهادة — الخطوة الأهم بكل الدليل

    هون بالضبط وقفت آخر مرة. لا تكمّل لأي إشي قبل ما تتأكد إنه الشهادة حقيقية (Let's Encrypt) وتطابق اسم السيرفر — مو ذاتية التوقيع ولا لدومين تاني.

    الدرس الذهبي: بدون شهادة تطابق اسم السيرفر، IMAP/SMTP ما بيشتغلوا

    لو الشهادة موجودة بس لاسم غلط، كل عميل (Thunderbird، Gmail، الموبايل) بيرفض الاتصال ويقول Wrong Site / NET::ERR_CERT_COMMON_NAME_INVALID. المنافذ بتكون مفتوحة والسيرفر شغّال — بس التشفير بيفشل لأنه الاسم مش مطابق.

    سبب المشكلة الشائع (صار فعلياً): بخانة Default Email Domain بالمعالج تحط دومين (مثلاً ehabkahwati.com)، فStalwart بياخد شهادة له ويجعلها الافتراضية — بس اسم السيرفر mail.techkahwa.net ما بياخد شهادة لأنه مش مسجّل كدومين، فبيقع على الافتراضي الغلط.

    الوقاية (الأسهل): خلّي اسم السيرفر من نفس دومينك الأساسي

    لو اخترت اسم السيرفر يكون subdomain من دومينك الأساسي (مثلاً اسم السيرفر mail.techkahwa.net ودومينك الأساسي techkahwa.net)، الشهادة التلقائية بتغطّيه ضمناً وما بتواجه المشكلة أصلاً.

    العلاج (لو اخترت دومين مختلف كافتراضي)

    لو صار عندك المشكلة (اسم السيرفر على دومين، والافتراضي على دومين تاني)، صلّحها هيك:

    • ١) Directory → Domains → Create domain → أضف اسم السيرفر نفسه كدومين (مثلاً mail.techkahwa.net). هيك Stalwart بيطلبله شهادة ACME. (ما بتحتاج حسابات عليه.)
    • ٢) على هالدومين خلّي Certificate Management = Automatic مربوط بمزوّد Let's Encrypt.
    • ٣) Settings → TLS → صحّح Default Certificate ليأشّر على شهادة اسم السيرفر (مش الدومين التاني).
    • ٤) systemctl restart stalwart واستنى دقيقة–دقيقتين.

    استنى دقيقتين بعد إعادة التشغيل (الشهادة بتاخد وقت)، بعدين:

    تحقق من الشهادة (من الماك أو أي جهاز)
    openssl s_client -connect mail.techkahwa.net:443 -servername mail.techkahwa.net \
  </dev/null 2>/dev/null | openssl x509 -noout -issuer -subject -dates
# issuer= لازم يحتوي "Let's Encrypt"
# subject= لازم يطابق اسم سيرفرك (techkahwa) — مش دومين تاني!

# وتأكد المنافذ فتحت بعد المعالج (شغّله على السيرفر):
ss -tlnp | grep -E ':443|:993|:465'
    لو فشلت — اقرأ سجل ACME (على السيرفر)
    journalctl -u stalwart --no-pager | grep -iE "acme|certificate|order|challenge|error" | tail -30
# ملاحظة: journalctl على السيرفر فقط — مش على الماك
    نقطة فحص ٧ — بوابة حاسمة
    • issuer= فيه Let's Encrypt (مو self-signed).
    • subject= يطابق اسم سيرفرك mail.techkahwa.net — مو دومين تاني.
    • المنافذ 443 و993 و465 كلها LISTEN.
    • https://mail.techkahwa.net بيفتح بدون تحذير، وThunderbird بيتصل بدون "Add Exception".
    إذا subject= دومين غلط: نفّذ «العلاج» فوق (أضف اسم السيرفر كدومين + صحّح Default Certificate). إذا self-signed: راجع سجل ACME والتوكن. لا تكرّر restart كتير حتى ما تنحظر من Let's Encrypt.

    فعّل 587 + افحص المنافذ فعلياً (بعد المعالج)

    الآن بس — بعد ما كمّلت المعالج وضبطت Cloudflare/DNS وأعدت التشغيل — صار Stalwart فاتح الـlisteners فعلاً، فبنقدر نفعّل 587 ونفحص المنافذ. قبل هالنقطة الفحص بيطلّع نتائج مضلّلة.

    أ) فعّل منفذ 587 (اختياري — لعملاء STARTTLS)

    Stalwart بيجي مفعّل عليه 465/993 (implicit TLS) بس، ومو مشغّل 587/143 (STARTTLS). لو عندك عملاء بتتوقع 587 (شائع بالشركات)، فعّله:

    باللوحة: Settings → Network → Listeners → Create
    • ID: submission · Protocol: smtp
    • Bind: [::]:587
    • Implicit TLS: OFF (587 بيستعمل STARTTLS، مو تشفير مباشر)
    • احفظ
    أعد التشغيل وتأكد
    systemctl restart stalwart
ss -tlnp | grep ':587'     # لازم يظهر LISTEN

    وتأكد إنه مفتوح بالجدار كمان (كان محضّر بخطوة ٣): قاعدة Inbound TCP 587.

    587 ≠ حل مشكلة الإرسال لبرّا

    587 هو باب دخول رسائل عملائك لسيرفرك (وارد). الإرسال لبرّا (لـGmail) بيمرّ على 25 الصادر — فتفعيل 587 ما بيغيّر شي بموضوع الحجب. الاتنين منفصلين.

    ب) افحص كل منفذ فعلياً (مو تفترض)

    اختبر المنافذ من برّا السيرفر (زي ما العالم بيشوفك). الأمر حسب جهازك:

    من macOS (مجرَّب وشغّال)
    for p in 25 143 443 465 587 993 995 4190; do
  nc -z -G 5 mail.techkahwa.net $p >/dev/null 2>&1 \
    && echo "$p OPEN" || echo "$p CLOSED"
done
    من جهاز/سيرفر لينكس
    for p in 25 143 443 465 587 993 995 4190; do
  timeout 5 bash -c "echo > /dev/tcp/mail.techkahwa.net/$p" 2>/dev/null \
    && echo "$p OPEN" || echo "$p CLOSED"
done
    فرق الأوامر — درس مهم

    أمر timeout مو موجود على macOS، فأمر لينكس بيفشل عالماك بصمت وبيطلّع كل شي CLOSED غلط. عالماك استعمل nc -z -G 5 (الـ-G = مهلة الاتصال). أو استعمل موقع check-host.net/check-tcp من أي جهاز.

    النتيجة المتوقعة لكل منفذ — هيك بتعرف شو طبيعي وشو مشكلة:

    منفذالمتوقع من برّاالتفسير
    443 · 465 · 993 · 4190OPENالأساسيات: لوحة/webmail، إرسال عملائك، استقبالهم، فلاتر Sieve.
    587OPEN لو فعّلته فوقإرسال عملائك بـSTARTTLS.
    995OPEN إذا بدك POP3ضروري لربط Gmail (بيسحب البريد عبر POP3 على 995). لو CLOSED والسيرفر ماسكه → القاعدة ناقصة بجدار Hetzner.
    25لا يُفحص من شبكة منزليةمزوّدو الإنترنت المنزلي بيحجبوا 25 من شبكتهم — فحصه من البيت بيطلّع CLOSED دايماً حتى لو سليم. الدليل الحقيقي: استقبالك للبريد. أو افحصه من check-host.net.
    143CLOSED مقبولIMAP STARTTLS — 993 بيغني عنه. تفعيله اختياري من Listeners.
    نقطة فحص ٧+ — المنافذ فعلياً
    • 443 و465 و993 و4190 كلهن OPEN من برّا.
    • 587 = OPEN لو فعّلته، و995 = OPEN لو بدك ربط Gmail.
    • لو استعملت جدارين: القواعد متطابقة، والـOutbound فاضي.
    CLOSED من برّا بس OPEN من السيرفر (ss -tlnp): الحاجب جدار Hetzner — أضف القاعدة Inbound. CLOSED من الاتنين: Stalwart مو ماسكه — Settings → Listeners (طبيعي لـ143؛ ولـ587 فعّله فوق). منفذ 25 لا يُحكم عليه من شبكة منزلية.

    فحص سجلات دومين البريد

    باللوحة

    سجل MX مفقود — لازم تضيفه يدوي

    لاحظنا بإعدادك إنه Stalwart نشر كل السجلات إلا الـMX. وبدون MX، ما حدا بيعرف يوصل بريد دومينك لسيرفرك. أضفه يدوي بـCloudflare:

    • Type: MX
    • Name: @ (أو techkahwa.net)
    • Mail server: mail.techkahwa.net
    • Priority: 10 · Proxy: DNS only

    كرّر إضافة الـMX يدوي لكل دومين تضيفه.

    تحقق من سجلات الدومين
    dig +short MX techkahwa.net                 # → 10 mail.techkahwa.net (لو فاضي، أضفه يدوي)
dig +short TXT techkahwa.net                # SPF: v=spf1 ...
dig +short TXT _dmarc.techkahwa.net         # DMARC
# DKIM: خذ اسم الـselector من اللوحة ثم:
dig +short TXT SELECTOR._domainkey.techkahwa.net
    نقطة فحص ٨
    • الـMX بيأشّر على mail.techkahwa.net.
    • الـSPF وDMARC وDKIM كلهن موجودين.
    • كل هالسجلات بـCloudflare رمادية.
    الـMX دايماً بتضيفه يدوي (Stalwart ما بينشره). إذا سجل تاني ناقص: تأكد التوكن All zones وإنه الدومين مضاف باللوحة، وأعد الفحص بعد دقيقة (TTL). إذا ضل، أضفه يدوي من القيمة اللي باللوحة.

    اختبار الإرسال والاستقبال (الإثبات الحقيقي)

    هون الاختبار الفعلي. ما نفترض إشي — نختبر منفذ 25 الصادر أول (بوابة حاسمة)، بعدين نرسل من أكتر من منفذ.

    البوابة الحاسمة: منفذ 25 الصادر

    منفذ 25 الصادر هو شريان الحياة — بدونه مستحيل ترسل لأي سيرفر خارجي (Gmail/Outlook)، مهما كان باقي الإعداد صح. اختبره أول:

    اختبار منفذ 25 الصادر
    timeout 5 nc -zv gmail-smtp-in.l.google.com 25; echo "exit: $?"
# succeeded / exit: 0  → المنفذ مفتوح، كمّل
# timed out / exit: 124 → محجوب، لازم تطلب فك الحجب (تحت)
    إذا كان محجوباً (متوقّع على Hetzner الجديد) — اطلب فك الحجب

    Hetzner بيحجب 25 · 465 · 587 الصادرة تلقائياً كمكافحة سبام. مو من ufw ولا Cloud Firewall — بس بطلب رسمي عبر الرابط المباشر:

    • console.hetzner.com/limits
    • Request change → Unblock Ports
    • اكتب الرسالة التالية:

    I'm running a legitimate self-hosted mail server (Stalwart) on IP 91.98.194.24 for my own domains. Please unblock outbound SMTP ports 25, 465, and 587. The server has correct reverse DNS (mail.techkahwa.net), SPF, DKIM, and DMARC. Thank you.

    بعض الحسابات بينفك تلقائياً خلال دقائق؛ غيرها بيراجعوه يدوي (لحد يوم). بعدها الرسائل العالقة بالطابور بتنطلق لحالها.

    نقطة فحص ١٠-أ — بوابة حاسمة
    • nc لمنفذ 25 رجّع succeeded / exit: 0.
    إذا timed out: منفذ 25 محجوب — قدّم طلب فك الحجب وانتظر قبل ما تكمّل اختبار الإرسال لبرّا. (الاستقبال والبريد الداخلي بيشتغلوا بالوقت الحالي.)

    اختبار الإرسال — من المنفذين (465 و587)

    نتأكد إنه الإرسال شغّال من كلا منفذي الإرسال، مو واحد بس:

    إرسال عبر 465 (SSL مباشر)
    apt -y install swaks
swaks --to YOUR_ADDRESS@gmail.com --from ehab@techkahwa.net \
      --server mail.techkahwa.net:465 --tls-on-connect \
      --auth LOGIN --auth-user ehab@techkahwa.net
    إرسال عبر 587 (STARTTLS)
    swaks --to YOUR_ADDRESS@gmail.com --from ehab@techkahwa.net \
      --server mail.techkahwa.net:587 --tls \
      --auth LOGIN --auth-user ehab@techkahwa.net
    ملاحظات من التجربة الفعلية
    • Stalwart أحياناً بيفتح 465/993 (implicit TLS) بس، وبيترك 587/143 (STARTTLS) مطفي. لو 587 أعطى Connection refused، استعمل 465 — بيعمل نفس الشي. لتفعيل 587: Settings → Listeners.
    • ناتج 250 Message queued معناه السيرفر قبل الرسالة — بس التسليم الفعلي لبرّا بيعتمد على منفذ 25 (فوق). راقب الوصول الفعلي لـGmail.
    • لا تحط كلمة السر بسطر الأمر — swaks بيسألك ياها بأمان. (وإذا انكشفت، غيّرها فوراً.)

    تحقق نهائي من التسليم والسمعة

    • افتح الرسالة بـGmail → Show original → لازم: SPF=PASS · DKIM=PASS · DMARC=PASS.
    • ردّ من Gmail على العنوان → لازم توصل لصندوقك (استقبال).
    • أرسل لـmail-tester.com — الهدف 10/10. وافحص MXToolbox Blacklist لـIP السيرفر.
    نقطة فحص ١٠-ب
    • وصلت لـGmail بالإنبوكس مو السبام، والتلاتة PASS.
    • الإرسال نجح من 465 (و587 إذا مفعّل).
    • الرد من Gmail وصل لصندوقك، وmail-tester = 10/10، والـIP مو على blacklist.
    إذا السيرفر قبل (queued) بس ما وصلت Gmail: غالباً منفذ 25 الصادر — راجع البوابة الحاسمة فوق. إذا راحت سبام: تأكد PTR + SPF/DKIM/DMARC وابدأ warmup (خطوة ١٢).

    الطبقة ٤ — استبعد جدرانك (قبل ما تتهم المزوّد)

    ٤-أ: ufw الصادر — الافتراضي بيسمح بكل الصادر، بس تأكد:

    سياسة ufw الصادرة
    ufw status verbose | grep -i default
# لازم: default: deny (incoming), allow (outgoing)
# لو outgoing = deny → هاد الحاجب! صلّحه:
#   ufw default allow outgoing && ufw reload

    ٤-ب: Hetzner Cloud Firewall — من الـConsole → Firewalls → تبويب Outbound:

    • فاضي (Nothing) = كل الصادر مسموح ✅ — مو هو الحاجب.
    • فيه قواعد = بس المذكور مسموح ⚠️ — إذا 25 مو مذكور، هاد الحاجب. أضف قاعدة TCP 25 صادرة أو احذف كل قواعد الـOutbound.
    الحالةالاستنتاج
    ufw outgoing = allow · الـOutbound فاضي · و25 لسا BLOCKEDجدرانك بريئة — الحجب upstream. → الطبقة ٥
    لقيت قاعدة حاجبة بأي جدارصلّحها وأعد اختبار الطبقة ٣.

    الطبقة ٥ — الاستنتاج النهائي: الحجب من Hetzner (upstream)

    الدليل القاطع مكتمل

    الصادر عموماً شغّال (443 OPEN) + جدرانك سامحة + منفذ 25 timeout لأكتر من هدف = Hetzner حاجب منفذ 25 على مستوى الشبكة. هاد سلوكهم الافتراضي للحسابات الجديدة (مكافحة سبام)، وما بينحل بأي إعداد عندك.

    • الطلب: console.hetzner.com/limits → Request change → Unblock Ports → اشرح إنه سيرفر بريد شرعي بـPTR/SPF/DKIM/DMARC مضبوطين. (بينفك تلقائياً خلال دقائق لبعض الحسابات.)
    • ملاحظة: الحسابات الجديدة ما بيفكّولها قبل أول فاتورة مدفوعة.
    • التحقق بعد الفك: أعد أمر الطبقة ٣ → لازم 25 OUT: OPEN. الرسائل العالقة بالطابور بتنطلق لحالها (أو أعد جدولتها من لوحة Queues).
    نقطة الفحص النهائية للتشخيص
    • حدّدت الطبقة اللي فشلت بدليل من أمر، مو بتخمين.
    • إذا وصلت للطبقة ٥: قدّمت الطلب، وبعد الفك 25 OUT: OPEN والرسالة وصلت Gmail فعلياً.
    قاعدة: لا تعيد تشغيل الخدمة أو تغيّر إعدادات عشوائياً وأنت عم تشخّص — كل تغيير غير موثّق بيخرّب الدليل وبيرجّعك لنقطة الصفر.

    التسلسل المنطقي الكامل للإعداد (خلاصة عملية)

    هاد الترتيب الصح مبني على التجربة الفعلية — نقطة الفحص الأساسية إنك ما تقدر تفحص المنافذ إلا بعد ما تكمّل المعالج وتعيد التشغيل، لأنه قبلها Stalwart ما بيكون فاتح كل الـlisteners.

    1. سيرفر Hetzner + PTR (خطوة ١)
    2. سجل A + رمادي بـCloudflare (خطوة ٢)
    3. الجدار الناري — افتح كل المنافذ Inbound (خطوة ٣). ملاحظة: الفتح بالجدار مش معناه المنفذ شغّال — لسا Stalwart ما فتحه
    4. تنصيب Stalwart + حساب طوارئ (خطوة ٤)
    5. تحقّق توكن Cloudflare (خطوة ٥)
    6. أكمل المعالج (٥ خطوات): Default Domain + ربط Cloudflare DNS بالتوكن (خطوة ٦)
    7. أعد تشغيل النظام — systemctl restart stalwart ← هون بس بتفتح كل الـlisteners فعلياً
    8. الآن افحص المنافذ — نزّل الإيميل على جهازك أول وكمّل إعداده، وبعدها تفحص المنافذ من برّا (خطوة ٣ — قسم الفحص)
    9. افحص الشهادة — لازم تطابق اسم السيرفر (خطوة ٧)
    10. سجلات DNS: تأكد من نشرها، وأضف MX يدوي (مش بينضاف أول مرة — خطوة ٨)
    11. فعّل 587 يدوياً (مش بينضاف تلقائياً) + restart مرة تانية
    12. أنشئ الحسابات + افتح البريد + اختبر الإرسال (خطوات ٩–١٠)
    القاعدة الذهبية للفحص

    ترتيب الفحص الصح: أكمل المعالج → restart → نزّل الإيميل واعمل setup على جهازك → بعدها افحص المنافذ من برّا. لو فحصت قبل الـrestart، رح تلاقي منافذ «مسكّرة» وهي فعلياً مش مفتوحة بعد.

    درس مهم: رسائل عالقة رغم إنه كل شي مضبوط

    لو الرسائل عالقة بالطابور (Next Retry) ومنفذ 25 الصادر مفتوح والإعداد سليم — معالج الطابور ممكن يكون علِق بعد تعديلات/restarts متتالية. الحل الأول: systemctl restart stalwart. غالباً بينطلق الطابور فوراً. وتجنّب عدة restarts سريعة متتالية.

    عند إضافة دومين جديد — قائمة كاملة

    كل مرة تضيف دومين (من الـ٢٠)، اعمل هالخطوات بالترتيب وتأكد منها.

    1. Directory → Domains → Create domain → اكتب الدومين (مثلاً domain2.com).
    2. اضغط على الدومين → قسم DNS → خلّي DNS Management = Automatic، والـDNS Server = مزوّدك (مثلاً ehab / Cloudflare اللي ضبطته).
    3. قسم Record Types: تأكد إنه مفعّل عليها كلها (Autoconfig, Autodiscover, CAA, DKIM, DMARC, MTA-STS, MX, SPF, SRV, TLS-RPT) — هاي اللي بتخلّي الأجهزة تكتشف الإعدادات لحالها.
    4. قسم TLS → Certificate Management = ACME، والـProvider = Let's Encrypt.
    5. أضف MX يدوياً بـCloudflare — لأنه Stalwart ما بينشره أول مرة: Type MX · Name @ · Server mail.techkahwa.net · Priority 10 · رمادي.
    6. أنشئ حسابات الدومين من Directory → Accounts.
    7. تأكد كل السجلات رمادية (DNS only) بـCloudflare.
    مصطلحات الشاشة (من التجربة)
    • Zone Origin (اختياري): لو الدومين subdomain (sub.example.com) وسجلاته تُدار بزون example.com، حط الأصل هنا. للدومين العادي اتركه فاضي.
    • Additional Hostnames (SANs) بقسم TLS: أسماء إضافية للشهادة (مثل mta-sts, autoconfig). اتركه فاضي لتاخد wildcard أو الافتراضي.
    تحقّق سجلات الدومين الجديد
    dig +short MX domain2.com                    # → 10 mail.techkahwa.net (أضفه يدوي لو فاضي)
dig +short TXT domain2.com                   # SPF
dig +short TXT _dmarc.domain2.com            # DMARC
dig +short SRV _imaps._tcp.domain2.com       # اكتشاف تلقائي للأجهزة
    نقطة فحص ١٤ — لكل دومين جديد
    • DNS Management = Automatic، وTLS = ACME.
    • كل Record Types مفعّلة.
    • MX مضاف يدوياً ويأشّر على اسم السيرفر.
    • كل السجلات رمادية، والاختبار على mail-tester.com = 10/10.
